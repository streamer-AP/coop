# OMAO v2 技术架构决策文档

## 一、项目概述

OMAO 是一款与蓝牙硬件设备配合使用的 APP，核心功能围绕 Live2D 角色交互、音频播放（共鸣）、蓝牙设备控制三大模块展开。包含三个角色：Orven、Solan、Varyn。

## 二、技术栈选型

| 层面 | 技术选型 | 说明 |
|------|---------|------|
| 主框架 | Flutter | 共鸣、控制器、主页、账号等全部非 Live2D 页面 |
| Live2D 渲染 | Unity | 剧情主页、剧情内页的 Live2D 模型交互 |
| Flutter-Unity 集成 | 原生容器切换 | Unity 和 Flutter 运行在独立原生页面，通过原生层切换 |
| 通信协议 | JSON | 消息量小，开发调试优先，后续可局部替换 Protobuf |
| 状态管理 | Riverpod | 声明式、编译安全、自动依赖追踪 |
| 本地数据库 | SQLite + drift | 关系型数据，条目-合集多对多、播放列表排序 |
| 共鸣音频引擎 | just_audio + audio_service | 后台播放 + 系统通知栏/锁屏控件集成 |
| 蓝牙 | Flutter BLE 统一管理 | 所有蓝牙信号统一由 Flutter 侧仲裁和发送 |
| 后端 API | RESTful | 简单直接，适合 CRUD 为主的业务 |
| 字幕翻译 | 阿里云翻译 | 阿里云生态内统一管理 |
| 实名认证 | 阿里云实名认证 | 身份证二要素验证 |
| 资源解密 | AES + 本地内置密钥 | 混淆保护，离线可用 |
| 最低版本 | Android 10+ / iOS 15+ | 覆盖约 85% 设备 |

## 三、架构决策详情

### 3.1 Flutter-Unity 集成方案：原生容器切换

- Unity 和 Flutter 各为独立 Activity(Android) / ViewController(iOS)
- 进入 Live2D 场景时切换到 Unity 页面，返回时切回 Flutter 页面
- 隔离性好，两侧互不干扰
- 切换时可能有短暂过渡，需设计 loading 动画缓解

### 3.2 语音/字幕控制：Unity 侧

- Unity 负责 Live2D 动画 + 语音播放 + 字幕渲染
- 剧情场景下字幕与动画强绑定，Unity 内部控制时序延迟最低
- 原生容器切换方案下 Flutter UI 无法叠加在 Unity 页面上
- 共鸣模块字幕在 Flutter 侧独立渲染，两套互不干扰

### 3.3 蓝牙信号传递路径：Unity → Native → Flutter BLE

```
剧情场景: Unity(信号数据) → Native Bridge → Flutter(BLE Service) → 蓝牙设备
共鸣场景: Flutter(音频解析) → Flutter(BLE Service) → 蓝牙设备
控制器场景: Flutter(预设波形) → Flutter(BLE Service) → 蓝牙设备
```

- 蓝牙逻辑集中在 Flutter 侧，信号冲突（预设/同步共鸣/剧情）统一仲裁
- Unity 只负责发送信号数据，不直接操作蓝牙
- 链路多一跳延迟约 5-15ms，200ms 发信间隔下完全可接受

### 3.4 Unity 进程管理：常驻 + 延迟初始化

- App 启动时后台初始化 Unity 引擎，不加载模型资源
- 进入剧情时加载对应角色资源，切换体验流畅，避免冷启动 2-5s 黑屏
- 退出剧情场景时卸载模型资源（纹理/动画/音频），仅保留引擎空壳（约 50-80MB）
- 兜底策略：内存告警时销毁 Unity，下次进入重新启动

## 四、模块职责划分

### 4.1 Unity 侧职责（剧情主页 + 剧情内页）

- Live2D 模型加载 / 渲染 / 动画播放（moc3 + motion3.json）
- 语音音频播放及时序控制
- 字幕渲染（srt 解析 + 显示）
- HitArea 点击检测（5个区域）
- 段落 / 检查点状态管理
- 向原生层发送：蓝牙信号数据、动画状态、段落进度、权限查询

### 4.2 Flutter 侧职责

- 主页、消息、个人中心等全部 UI
- 共鸣播放器（just_audio + audio_service）
- 控制器页面及波形编辑
- 蓝牙 BLE 统一管理及信号冲突仲裁
- 权限体系（实名认证、页面权限码、设备绑定解锁）
- 用户状态管理（登录注册、剧情进度、云端配置同步）
- 日志记录及回传
- 资源下载 / 解密 / 缓存管理

### 4.3 原生桥接层职责

- Flutter ↔ 原生：MethodChannel / EventChannel
- Unity ↔ 原生：UnitySendMessage / Unity 原生插件
- 页面切换管理（Flutter ↔ Unity 容器切换）
- 生命周期协调（前后台切换、内存告警）

## 五、Flutter-Unity 通信协议

### 5.1 消息格式

采用 JSON，统一消息结构：

```json
{
  "type": "消息类型",
  "data": { ... }
}
```

### 5.2 Flutter → Unity 消息

| type | 说明 | data 字段 |
|------|------|----------|
| `enterScene` | 进入剧情场景 | characterId, sceneId, permissions[] |
| `exitScene` | 退出剧情场景 | - |
| `skipToCheckpoint` | 跳过到检查点 | checkpointId |
| `userProgress` | 同步用户进度 | completedSections[] |
| `bluetoothState` | 蓝牙连接状态 | connected, deviceModel |
| `bluetoothToggle` | 蓝牙开关控制 | enabled |

### 5.3 Unity → Flutter 消息

| type | 说明 | data 字段 |
|------|------|----------|
| `bluetoothSignal` | 蓝牙信号数据 | swing, vibration (0-100) |
| `sectionComplete` | 段落完成 | sectionId |
| `checkpointReached` | 到达检查点 | checkpointId, isEnding |
| `animationState` | 动画状态变化 | state (idle/playing/finished) |
| `requestExit` | 请求返回 Flutter | - |
| `storyComplete` | 整段剧情完成 | storyId |

## 六、蓝牙信号冲突状态机

三种信号源存在互斥关系，统一由 Flutter BLE Service 仲裁：

```
信号源优先级（高 → 低）：
  剧情信号 > 同步共鸣 > 预设波形

切换规则：
  ┌─────────────────────────────────────────────────────┐
  │ 当前状态        触发事件              结果           │
  ├─────────────────────────────────────────────────────┤
  │ 预设波形    →  进入航线页         →  打断，信号停止  │
  │ 预设波形    →  进入剧情主页       →  不影响          │
  │ 预设波形    →  开启同步共鸣强度   →  切换到同步共鸣  │
  │ 同步共鸣    →  进入剧情主页       →  打断，信号停止  │
  │ 同步共鸣    →  音频暂停           →  信号暂停        │
  │ 同步共鸣    →  切换预设波形强度   →  切换到预设波形  │
  │ 剧情信号    →  退出剧情           →  信号停止        │
  │ 任意状态    →  蓝牙断开           →  双电机强度归零  │
  │ 任意状态    →  App 进入后台       →  继续发送信号    │
  └─────────────────────────────────────────────────────┘
```

## 七、开发阶段规划

```
Phase 0: 技术验证
  ├── Flutter-Unity 原生容器切换 PoC（Android + iOS）
  ├── Unity Live2D SDK 加载 moc3 模型 + HitArea 检测
  ├── Flutter BLE 200ms 间隔稳定发信验证
  └── 三路同步（Unity 动画 + 音频 + 蓝牙信号）可行性

Phase 1: 基础框架
  ├── Flutter 项目结构搭建（模块划分、路由、状态管理）
  ├── 原生桥接层（MethodChannel + Unity 通信）
  ├── 账号体系（注册/登录/实名认证）
  └── 权限体系框架

Phase 2: 控制器模块
  ├── 蓝牙扫描/连接/断开
  ├── 波形控制（双电机独立控制）
  ├── 常用波形配置（12格 × 摇摆/震动）
  ├── 自定义波形编辑器（最多18个，32s，线性插值）
  └── 账号云端配置同步

Phase 3: 共鸣模块
  ├── 音频播放器（just_audio + audio_service）
  ├── 文件/压缩包导入 + 视频提取音频
  ├── 字幕/封面/台本自动匹配算法
  ├── 合集管理 + 播放列表（三种循环模式）
  ├── 字幕跟随 + 翻译（中英日韩）
  ├── 同步共鸣（音频解析 → 蓝牙信号）
  └── 后台播放 + 系统播放控件

Phase 4: 剧情模块
  ├── Unity 剧情主页（Live2D 交互 + 待机/主动交互）
  ├── 剧情内页（段落 + 检查点 + 字幕 + 音频）
  ├── 剧情蓝牙信号同步
  ├── 剧情进度管理 + 权限解锁（逐段解锁）
  └── 回忆页（溯回星图）

Phase 5: 集成与打磨
  ├── 蓝牙信号冲突状态机联调
  ├── 权限体系全链路（实名认证 + 设备绑定 + 页面权限）
  ├── 日志记录 + 幂等回传
  ├── 资源下载/解密/缓存策略
  └── 性能优化 + 内存管理
```

### 9.2 Feature 内部分层

每个 feature 模块统一采用以下四层结构：

```
feature_name/
├── data/              # Repository 实现、API、本地数据源
├── domain/            # 纯 Dart 模型 + Repository 抽象接口（零外部依赖）
│   └── models/
├── application/       # Service 业务逻辑 + providers/
└── presentation/      # screens/ + widgets/
```

### 9.3 数据库设计（单数据库 + 多表 + DAO 分域）

核心表：
- **resonance**: audio_entries, audio_collections, entry_collection_cross_ref, playlists, playlist_items, subtitles, signal_files
- **controller**: waveform_presets, custom_waveforms, waveform_keyframes
- **story**: story_progresses, story_checkpoints
- **user**: user_preferences, device_bindings, cached_permissions

### 9.4 Riverpod Provider 组织

- **keepAlive**: appDatabase, dio, auth, bleConnectionManager, bleSignalArbitrator, unityBridge, permission
- **autoDispose**: player, playlist, subtitle, waveformEditor, storyProgress, collectionDetail, profile
- 使用 riverpod_generator（@riverpod / @Riverpod 注解）
- Provider 文件放在各自模块的 `application/providers/` 下

waveform_player_service  ─┼─► ble_signal_arbitrator ─► ble_signal_sender ─► BLE Device
